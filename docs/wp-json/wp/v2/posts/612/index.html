With the new year now behind us, to kick start 2014 Agile Staffordshire will be going back to basics with a twist. A key technique in agile development is Test Driven Development (TDD), a difficult skill to master yet often practiced through simple Kata exercises. For the uninitiated, Katas are a reference to martial arts where students repeatably\u00a0practice a set of moves in order to better learn and understand. Programming Katas are similar short 10-30 minute challenges where the goal is the same but each attempt can lead you to experience a different journey.<\/p>\n

This month we will be pairing together to do the String Calculator by Roy Osherove<\/a>\u00a0but with a constraint of your choosing. Kata rules are below, as are a selection of constraints but feel free to choose your own. There is no set language, but you will need one with a testing framework for TDD.<\/p>\n

String Calculator<\/h2>\n

The following is a TDD Kata- an exercise in coding, refactoring and test-first, that you should apply daily for at least 15 minutes (I do 30).<\/p>\n

Before you start:<\/h2>\n
    \n
  • Try not to read ahead.<\/li>\n
  • Do one task at a time.\u00a0The trick is to learn to work incrementally.<\/li>\n
  • Make sure you only test for\u00a0correct inputs<\/strong>. there is no need to test for invalid inputs for this kata<\/li>\n<\/ul>\n

    String Calculator<\/strong><\/p>\n

      \n
    1. Create a simple String calculator with a method\u00a0int Add(string numbers)<\/strong>\n
        \n
      1. The method can take 0, 1 or 2 numbers, and will return their sum (for an empty string it will return 0) for example\u00a0\u201c\u201d or \u201c1\u201d or \u201c1,2\u201d<\/strong><\/li>\n
      2. Start with the simplest test case of an empty string and move to 1 and two numbers<\/li>\n
      3. Remember to solve things as simply as possible so that you force yourself to write tests you did not think about<\/li>\n
      4. Remember to refactor after each passing test<\/li>\n<\/ol>\n<\/li>\n
      5. Allow the Add method to handle an unknown amount of numbers<\/li>\n
      6. Allow the Add method to handle new lines between numbers (instead of commas).\n
          \n
        1. the following input is ok:\u00a0 \u201c1\\n2,3\u201d\u00a0 (will equal 6)<\/li>\n
        2. the following input is NOT\u00a0ok:\u00a0 \u201c1,\\n\u201d (not need to prove it – just clarifying)<\/li>\n<\/ol>\n<\/li>\n
        3. Support different delimiters<\/strong>\n
            \n
          1. to change a delimiter, the beginning of the string will contain a separate line that looks like this:\u00a0\u00a0 \u201c\/\/[delimiter]\\n[numbers\u2026]\u201d for example \u201c\/\/;\\n1;2\u201d should return three where the default delimiter is \u2018;\u2019 .<\/li>\n
          2. the first line is optional. all existing scenarios should still be supported<\/li>\n<\/ol>\n<\/li>\n
          3. Calling Add with a negative number will throw an exception \u201cnegatives not allowed\u201d – and the negative that was passed.if there are multiple negatives, show all of them in the exception message
            \n
            \n

            stop here if you are a beginner<\/strong>. Continue if you can finish the steps so far in less than 30 minutes.<\/p>\n

            \n<\/li>\n
          4. Numbers bigger than 1000 should be ignored, so adding 2 + 1001 \u00a0= 2<\/li>\n
          5. Delimiters can be of any length with the following format:\u00a0 \u201c\/\/[delimiter]\\n\u201d for example: \u201c\/\/[***]\\n1***2***3\u201d should return 6<\/li>\n
          6. Allow multiple delimiters like this:\u00a0 \u201c\/\/[delim1][delim2]\\n\u201d for example \u201c\/\/[*][%]\\n1*2%3\u201d should return 6.<\/li>\n
          7. make sure you can also handle multiple delimiters with length longer than one char<\/li>\n<\/ol>\n

            <\/h2>\n

            Kata Constraints<\/h2>\n

            Select ONE<\/span> constraint from below.<\/p>\n

              \n
            • No If, Else, Switch Statements<\/li>\n
            • No mouse<\/li>\n
            • No Classes<\/li>\n
            • Use a functional language<\/li>\n
            • No Mutable State<\/li>\n
            • No Getters\/Setters<\/li>\n
            • No IDE<\/li>\n
            • 2 Refactorings after each passing test<\/li>\n
            • Single word method names<\/li>\n
            • Or make one up yourself…<\/li>\n<\/ul>\n

              Many of the constraints above which affect the design of code, work towards improving object calisthenics. More on this can be found here.<\/a><\/p>\n

              The goal of the session is not to complete the Kata (you may even restart several times)<\/em> but to learn through the experience of your selected constraint.<\/p>\n

              This will also be a special month, as it will be the last regular attendance of one of Agile Staffordshire’s founding members. Neil Kidd will be moving on to a new position at 7Digital<\/a>, so all please wish him the best.<\/p>\n

              Hope to see you there.
